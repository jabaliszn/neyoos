Part J Visual Proof Rule

This mockup documents the repaired J.25 standard for visual proof in this sandboxed environment.

Rule: Every visual J feature must have a visual proof artifact — a real screenshot where capture is possible, otherwise a labelled static HTML mockup. Fake screenshots are never allowed.

Accepted proof

✓ Real screenshot

Use when browser capture works and the real UI can be shown honestly.

✓ Labelled static HTML mockup

Use when screenshot capture is blocked by the environment but the UI still needs visual review.

Never allowed

✗ Fake screenshot

Do not fabricate a browser capture or claim one exists when it does not.

~ Partial visual evidence is acceptable only when clearly labelled

The founder must always know whether they are seeing a real screenshot or a mockup.

Why this repair was needed

The older J.25 wording said every visual J feature required a screenshot. That was too rigid for this sandbox because browser capture is sometimes unavailable. The repaired wording matches real practice and keeps the rule honest.

Future-chat guidance

When continuing Part J in another chat:

  • Prefer a real screenshot first.
  • If impossible here, produce a labelled static mockup.
  • Record the proof path in the checklist and context anchor.
  • Never blur the difference between real and mocked visuals.
