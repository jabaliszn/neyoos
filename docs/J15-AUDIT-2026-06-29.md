# J.15 Audit — Modular Report Builder
Date: 2026-06-29
Status: AUDIT ONLY (before any repair)

## Checklist being audited
Section: `J.15 — Modular Report Builder`

Lines under audit:
1. Create no-code report-template engine for CBC, internal reports, competency reports, portfolio reports, pathway reports and custom reports.
2. Reuse Document Design engine from I.42 where possible.
3. Support sections: marks, competencies, attendance, behavior, talent, portfolio, teacher comment, principal comment, QR verification.
4. School can choose report layout without code.
5. Report PDFs must be print-perfect and Powered by NEYO.
6. Report template changes are audit logged.

## What exists in the repo
### Real pieces found
- DB model exists: `ReportTemplate` with `name`, `description`, `isDefault`, `sectionsJson`.
- Validation exists: `src/lib/validations/report-builder.ts`
- Service exists: `src/lib/services/report-template.service.ts`
- API exists: `src/app/api/academics/report-templates/route.ts`
- UI exists and is mounted in Academics tab:
  - `src/components/academics/report-builder.tsx`
  - mounted through `src/components/academics/academics-client.tsx`
- Existing report-card PDF exists:
  - `src/lib/documents/report-card-pdf.tsx`
- Existing branded PDF already includes `Powered by NEYO · neyo.co.ke`

## What is NOT true yet
### 1) Not honestly a full report-generation engine
The current feature mostly stores template definitions. It does **not** yet drive a real dynamic PDF builder that reads a selected template and mounts sections from `sectionsJson` into generated reports.

What exists now:
- create/edit/delete/list templates
- choose section types in UI

What is missing:
- no runtime report generator using template sections
- no template-to-PDF rendering pipeline
- no real CBC / competency / portfolio / pathway / custom report output driven by selected template

Verdict: **partial only**.

### 2) I.42 Document Design engine is not reused here
There is a real document design service:
- `src/lib/services/document-design.service.ts`

But the J.15 template service and UI do not read or apply that engine in any visible J.15-specific way.

Verdict: **not built for J.15**.

### 3) Section support is only declarative, not fully functional
Validation allows these section names:
- `ACADEMIC_MARKS`
- `COMPETENCIES`
- `ATTENDANCE`
- `DISCIPLINE`
- `TALENTS`
- `PORTFOLIO`
- `TEACHER_REMARKS`
- `PRINCIPAL_REMARKS`
- `QR_VERIFICATION`

But this is only a list in schema/UI. The current repo does not show a dynamic report engine actually rendering all of them from a chosen template.

Also notable gap:
- checklist says **behavior**; implementation uses `DISCIPLINE`, which may be acceptable semantically, but it is still not fully wired to rendering.

Verdict: **partial only**.

### 4) Layout choice without code is real at template-definition level
This part is real:
- school can create templates in UI
- school can add/remove/reorder section definitions
- can set default template

But reorder appears only visually in the current editor; there is no robust drag-drop engine and no downstream generated PDF proving the chosen order affects output.

Verdict: **partial** rather than honest full-stack.

### 5) Print-perfect PDF requirement is not proven for modular builder
There is a real branded report-card PDF, but it is a fixed report-card PDF, not an honestly modular report-builder PDF.

So:
- `Powered by NEYO` exists in current report-card PDF ✅
- print-perfect modular-builder output is **not proven** ❌

Verdict: **partial only**.

### 6) Audit logging for template changes is missing
I checked `src/lib/services/report-template.service.ts`.
There is **no audit log write** on:
- create template
- update template
- delete template
- default-template switch

This fails the checklist line directly.

Verdict: **not built**.

## Code-level problems found
### Tenant safety bug
`src/lib/services/report-template.service.ts` uses `tenantDb()` directly but does **not** wrap the service methods in `withTenant(user.tenantId, ...)`.

That is the same class of bug we found in other Part J sections.

### UI prop bugs
`src/components/academics/report-builder.tsx` still uses invalid props:
- `<Badge variant="secondary">`
- `<Badge variant="outline">`
- `<Button variant="outline">`

This is not aligned with the actual shared UI component contracts in this repo.

### Test is dishonest / fragile
`scripts/j15-report-builder-test.ts`:
- directly inserts fixed template names with `db.reportTemplate.create(...)`
- not idempotent
- does not prove real full-stack behavior
- does not exercise API route, tenant-safe service flow, audit log, or template-driven PDF rendering
