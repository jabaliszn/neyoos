# J.20 — Future-Proof Configuration & Versioning — Repair Report
**Date:** 2026-06-29
**Action:** Option 1 — Fix J.20 fully (after honest audit + downgrade)
**Result:** ✅ Repaired to true full-stack and re-verified.

---

## 1. What J.20 promises (plain English)

| Promise | What it means for a school |
|---|---|
| Versioned curriculum | When CBC changes (e.g. CBC 2026 → CBC 2027), the school keeps the old version and creates a new one — nothing is overwritten. |
| Preview before switching | The school can look at a draft new curriculum before going live. |
| Migration assistant | Before publishing, the school sees exactly what will be added, removed or renamed, plus a warning about impact. |
| History keeps its version | An old report reprinted years later still uses the curriculum it was issued under. |
| Templates carry effective dates | Report templates and assessment types know which dates and curriculum version they belong to. |
| NEYO Ops templates | A school can import an official curriculum published by NEYO Ops (delivered in J.21). |

## 2. Honest state BEFORE repair (from the audit)

- Header was corrupted/duplicated, then downgraded to `PARTIAL`.
- 2 lines were genuinely real, the rest were partial or not built.
- The versioning service used `tenantDb()` **without** `withTenant(...)` (a tenant-isolation bug).
- No audit logging on draft / publish.
- The diff was shallow (added/removed only, no rename detection, no impact warning).
- `ReportTemplate` and `AssessmentType` had **no** version / effective-date fields wired through.
- No honest "historical report keeps its version" behaviour.
- The old test `scripts/j20-versioning-test.ts` was **dishonest** — it poked the DB directly with Prisma instead of exercising services / validation / UI.
- **Critical hidden bug discovered:** the shared `@/components/ui/dialog` primitive **did not exist on disk**, yet 10 components imported it (including the J.20 manager and previously-"completed" J.15 report builder). The UI could not compile.

## 3. What was fixed (full-stack, all 8 layers)

| Layer | Change |
|---|---|
| 1. DB & migrations | `AssessmentType.effectiveFrom/effectiveTo`; `ReportTemplate.version/effectiveFrom/effectiveTo/curriculumVersion`. Schema pushed to the SQLite dev DB. |
| 2. Security & validation | `assessmentTypeSchema` now validates effective dates (end ≥ start). `reportTemplateSchema` validates version + effective dates. API routes guard with `academics.view` (GET) and `academics.manage` (POST). |
| 3. Backend services | `curriculum-versioning.service.ts` rewritten to wrap every query in `withTenant(...)`, add audit logging for draft/publish, detect renames, and compute a migration-impact summary. `report-template.service.ts` now persists version/effective dates/curriculum version and resolves the correct historical template by date + version. `assessment.service.ts` persists effective dates on update. |
| 4. API endpoints | `GET/POST /api/curriculum/versions` fixed (double-wrapped `ok({data})` → `ok(data)`, correct status codes). Same wrapper bug fixed in `/api/curriculum/library`. |
| 5. UI components | `curriculum-version-manager.tsx` rebuilt: all invalid `Badge variant=` → `tone=`, invalid `Button variant="outline"` → `secondary`/`primary`. Richer diff dialog (added/removed/renamed + impact warning). |
| 6. Mounted surface | Still mounted in `academics-client.tsx` under the **Curriculum Versions** tab. Created the **missing** `src/components/ui/dialog.tsx` primitive so the surface (and 9 other components) actually compiles. |
| 7. Four UX states | Loading, error (with retry), empty, and data states all explicit in the manager. |
| 8. Kenyan seed data | Exercises the real seeded `karibu-high` CBC Kenya curriculum; test creates/cleans a `CBC 2027`-style draft and restores the seed. |
